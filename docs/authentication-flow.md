# กระบวนการยืนยันตัวตน (Authentication Flow)

เอกสารนี้อธิบายขั้นตอนการทำงานของระบบล็อคอินและการยืนยันตัวตนในโปรเจกต์ CAD Resource Center โดยจะเริ่มตั้งแต่ที่ผู้ใช้งานกดปุ่มล็อคอิน ไปจนถึงตอนที่ระบบสร้าง Session ให้เข้าใช้งานได้เลยครับ

ระบบของเราใช้หลักการ **OAuth2** ร่วมกับบริการของ **ThaID** เพื่อยืนยันตัวบุคคล และมีการตรวจสอบสิทธิ์การเป็นเจ้าหน้าที่ (Employee Verification) ผ่าน Backend ของกรมตรวจบัญชีสหกรณ์อีกชั้นนึงครับ เพื่อให้มั่นใจว่าระบบจะปลอดภัยและข้อมูลพนักงานอัปเดตอยู่เสมอ

---

## ขั้นตอนการทำงาน (Step-by-Step Flow)

### 1. ผู้ใช้กดปุ่ม Login
ผู้ใช้งานกดปุ่ม "ลงชื่อเข้าใช้ด้วย ThaID" ที่หน้าเว็บ ระบบจะทำการ Redirect ผู้ใช้ไปยังหน้าเว็บของ ThaID (ระบบ OAuth2 ของกรมการปกครอง) เพื่อให้ผู้ใช้ทำการยืนยันตัวตนผ่านแอปพลิเคชันหรือหน้าเว็บ ThaID

### 2. ThaID ส่ง Code กลับมา (Callback)
เมื่อผู้ใช้ยืนยันตัวตนกับ ThaID สำเร็จ ThaID จะส่งผู้ใช้กลับมาที่ระบบของเรา (ผ่าน URL Callback เช่น `/auth/callback`) พร้อมกับแนบ Authorization `code` มาใน URL  

### 3. แลก Code เป็นเลขบัตรประชาชน (PID)
ระบบ Frontend (Next.js) ของเราจะนำ `code` ที่ได้ไปส่งให้ฟังก์ชัน `getThaIDOwner(code)` ทำงาน เพื่อนำ code นี้ไปแลก (Exchange) กับ ThaID ให้กลายเป็น Access Token และดึงข้อมูลโปรไฟล์ของผู้ใช้งานกลับมา โดยสิ่งที่เราต้องการที่สุดในขั้นตอนนี้คือ **"เลขประจำตัวประชาชน (PID)"**

### 4. ตรวจสอบสิทธิ์กับ Backend (Employee Verification)
เมื่อได้เลขบัตรประชาชน (PID) มาแล้ว ระบบ Next.js จะส่ง PID นี้ไปถาม Backend API ของเรา (`POST /employee/verify`) 
> 💡 *ข้อมูลเพิ่มเติม:* ในการคุยกันระหว่าง Frontend server กับ Backend นี้ จะใช้ `AUTH_SECRET` ในการยืนยันตัวตนแบบ Server-to-Server เพื่อความปลอดภัย

การทำงานฝั่ง Backend เมื่อได้รับ PID จะมี 2 สเต็ปย่อยๆ คือ:
- **มองหาใน Local Database ก่อน:** Backend จะค้นหา PID นี้ในฐานข้อมูลระบบของเรา (`common_users`) เป็นอันดับแรก เพื่อให้ระบบยังสามารถล็อคอินได้รวดเร็วทันใจ แม้ว่าระบบ HR กลางของกรมฯ จะมีปัญหาหรือดาวน์อยู่ก็ตาม
- **ถ้าไม่เจอ สแกนหาจากระบบ HR (CWIS):** หากไม่พบข้อมูลใน Local Database ระบบ Backend จะวิ่งไปดึงข้อมูลจากฐานข้อมูลทรัพยากรบุคคล (HR) ของกรมฯ หากพบว่าเป็นเจ้าหน้าที่จริง ก็จะดึงข้อมูลมาเพิ่มใน Local Database ให้เลยแบบอัตโนมัติ

หลังจากนั้น Backend จะทำการสร้าง **Access Token** และส่งข้อมูลผู้ใช้พร้อมกับ Token นี้กลับมาให้ฝั่ง Frontend

### 5. ตรวจสอบสถานะการใช้งาน (Active Status)
เมื่อ Frontend ได้รับข้อมูลจาก Backend กลับมา ระบบ (ใน `auth.config.ts`) จะเช็คฟิลด์ `status` ก่อนเลย หากพบว่าผู้ใช้คนนี้มีสถานะเป็น `inactive` หรือถูกระงับการใช้งาน ระบบจะตัดจบและเด้งข้อความ "Access Denied" ทันที ไม่ให้เข้าใช้งานต่อครับ

### 6. สร้าง Session สรุปจบการล็อคอิน
หากผ่านด่านทั้งหมด ระบบ NextAuth (ฝั่ง Frontend) จะทำการนำข้อมูลที่ได้สร้างเป็น JWT และ Session ให้กับผู้ใช้งาน ประกอบไปด้วย:
- **ID:** ไอดีในระบบของเรา (ไม่ใช่เลขบัตรประชาชนแล้ว)
- **Name:** ชื่อ-นามสกุล
- **Role:** สิทธิ์การใช้งาน (`admin` หรือ `user`)
- **Access Token:** Token สำหรับนำไปใช้เรียก Backend API ในครั้งต่อๆ ไป

เพียงเท่านี้ ผู้ใช้งานก็สามารถเข้าใช้งาน CAD Resource Center ได้แล้วครับ! 🎉 โดยทุกครั้งที่หน้าเว็บมีการไปขอข้อมูลจาก Backend ทะลุเข้าไปดึงไฟล์หรือข้อมูลอื่นๆ ตัว Frontend ก็จะแนบ Access Token ของผู้ใช้คนนี้ไปด้วยทุกครั้งเพื่อเป็นการยืนยันตัวตนครับ

## สรุป Component ที่เกี่ยวข้อง
- **`auth.config.ts`**: จุดศูนย์กลางของระบบ Login ฝั่ง Next.js ทำหน้าที่ควบคุมเงื่อนไข จัดการ Token และ Session
- **`lib/thaid-service.ts`**: จัดการเรื่องการคุยกับระบบภายนอก (ThaID)
- **`lib/backend-api.ts`**: สื่อสารกับฝั่ง Backend ของเรา โดยเฉพาะเส้น `/employee/verify`
